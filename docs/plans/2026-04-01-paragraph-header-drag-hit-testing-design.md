# Paragraph Header Drag Hit Testing Design

## 背景

句子翻译面板在常驻状态下，原文正文区域已经可以通过原生 `NSWindow.performDrag` 触发窗口拖动，但原文标题行在重启后仍然存在“看似可拖、实际拖不动”的问题。

## 根因分析

当前标题行使用的是“底层拖拽层 + 上层 SwiftUI 内容”的叠层结构。这个方案依赖上层标题行容器和文字视图把鼠标事件透传到底层 `NSViewRepresentable`。在 SwiftUI 和 AppKit 混排时，这种透传并不稳定，尤其在窗口重新创建或视图层级重建后，更容易出现上层容器先命中、底层拖拽视图拿不到 `mouseDown` 的情况。

## 方案对比

### 方案 A：继续使用整行底层拖拽层

- 优点：代码改动少。
- 缺点：仍然依赖事件穿透，问题本质不变。

### 方案 B：把标题行拆成显式命中分区

- 左侧为独立的可拖区域，覆盖标题文字和空白区。
- 右侧 `复制` / `关闭` 按钮保持独立命中。
- 优点：命中关系明确，不再依赖层级穿透。
- 缺点：需要调整标题行布局结构。

## 选型

采用方案 B。

## UI / 交互约束

- 标题行中，`复制` 和 `关闭` 图标保持按钮行为。
- 除这两个图标外，标题文字与剩余空白区都用于拖动窗口。
- 拖动仍然使用原生 `window.performDrag(with:)`，保证跟手性，不回退到 SwiftUI `DragGesture`。
- 标题行相对上边框的距离保持不变，不影响现有贴边视觉。
